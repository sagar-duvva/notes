# **2. MODULE 1 ‚Äî INSTALLATION & INITIAL CONFIGURATION**

## **Concept Explanation**

Git is a distributed version control system.
You track changes locally and synchronize with remote servers like GitHub.

### **Diagram (text-described):**

```
[Working Directory] ‚Üí [Staging Area] ‚Üí [Local Repository] ‚Üí [Remote Repository]
     (edit files)        (add)            (commit)               (push/pull)
```

---

## **üß™ LAB 1: Install Git**

### **Windows**

1. Download from `git-scm.com`
2. Install with default settings (includes Git Bash)

### **macOS**

```bash
brew install git
```

### **Linux**

```bash
sudo apt install git    # Debian/Ubuntu
sudo dnf install git    # Fedora
```

---

## **üß™ LAB 2: Configure Git**

Run the following:

```bash
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
git config --global core.editor "code --wait"   # VS Code
```

### Verify:

```bash
git config --list
```

### Example Output:

```
user.name=John Doe
user.email=john@example.com
core.editor=code --wait
```

# **3. MODULE 2 ‚Äî REPOSITORIES (LOCAL & REMOTE)**

## **Concept Explanation**

* **Local repo** lives on your machine
* **Remote repo** lives on GitHub
* Git connects them via URLs (HTTPS or SSH)

---

## **üß™ LAB 3: Create a Local Repository**

```bash
mkdir myproject
cd myproject
git init
```

**Output:**

```
Initialized empty Git repository in /myproject/.git/
```

---

## **üß™ LAB 4: Connect to a Remote Repository**

1. Create a repo on GitHub (no README)
2. Run:

```bash
git remote add origin https://github.com/<username>/myproject.git
git remote -v
```

# **4. MODULE 3 ‚Äî STAGING, COMMITTING & HISTORY**

## **Concept Explanation**

Git tracks changes in three spaces:

1. **Working Directory** ‚Äì your files
2. **Staging Area** ‚Äì what will be committed
3. **Local Repository** ‚Äì committed history

---

## **üß™ LAB 5: Stage and Commit Changes**

### Step 1: Create a file

```bash
echo "Hello World" > hello.txt
```

### Step 2: Stage it

```bash
git add hello.txt
```

### Step 3: Commit

```bash
git commit -m "Add hello.txt"
```

---

## **üß™ LAB 6: Inspect Git History**

```bash
git log --oneline --graph --decorate
```

Example Output:

```
* f5a1b3c (HEAD -> main) Add hello.txt
```

# **5. MODULE 4 ‚Äî BRANCHING, MERGING & MERGE STRATEGIES**

## **Concept Explanation**

Branches allow isolated development.

### Text Diagram:

```
main ---A---B------M----D
             \----C---/
              feature/login
```

---

## **üß™ LAB 7: Create and Switch Branches**

```bash
git checkout -b feature1
```

Edit a file:

```bash
echo "Feature change" >> hello.txt
git add hello.txt
git commit -m "Add feature1 change"
```

---

## **üß™ LAB 8: Merge Branches**

Return to main:

```bash
git checkout main
git merge feature1
```

---

## **Merge Strategies Explained**

* **Fast-forward** (no conflicts)
* **3-way merge** (new merge commit)
* **Squash merge** (condenses history)

---

## **üß™ LAB 9: Intentional Merge Conflict**

1. Edit *hello.txt* differently on `main` and `feature1`
2. Attempt merge:

```bash
git merge feature1
```

You will see:

```
CONFLICT (content): Merge conflict in hello.txt
```

Resolve file manually, then:

```bash
git add hello.txt
git commit
```

# **6. MODULE 5 ‚Äî UNDOING CHANGES, RESET, REVERT & REFLOG**

## **Concept Explanations**

* **git checkout -- file** ‚Üí discard uncommitted changes
* **git reset** ‚Üí move branch pointer
* **git revert** ‚Üí creates a new commit that undoes another
* **git reflog** ‚Üí history of all HEAD movements

---

## **üß™ LAB 10: Practice Resetting**

### Restore file:

```bash
git restore hello.txt
```

### Soft reset (keep changes):

```bash
git reset --soft HEAD~1
```

### Hard reset (dangerous):

```bash
git reset --hard HEAD~1
```

---

## **üß™ LAB 11: Reflog**

```bash
git reflog
```

Example:

```
a13e213 HEAD@{1}: commit: Add feature1 change
bd1c821 HEAD@{2}: checkout: moving from main to feature1
```

# **7. MODULE 6 ‚Äî GITHUB REMOTE OPERATIONS**

## **üß™ LAB 12: Push Code to GitHub**

```bash
git push -u origin main
```

---

## **üß™ LAB 13: Fetch vs Pull**

```bash
git fetch origin
git pull origin main
```

**Explanation:**

* **fetch** downloads but does not merge
* **pull** = fetch + merge

# **8. MODULE 7 ‚Äî PULL REQUESTS, REVIEWS & COLLABORATION**

## **Concept Explanation**

Pull Requests (PRs) allow team members to:

* Review code
* Discuss changes
* Run CI
* Merge into protected branches

---

## **üß™ LAB 14: Create a Pull Request**

1. Push a feature branch:

```bash
git push -u origin feature1
```

2. Open GitHub ‚Üí Create Pull Request
3. Add reviewers
4. Perform code review
5. Merge via "Squash", "Rebase", or "Merge commit"

---

## **Realistic Team Conflict Scenario**

* Developer A edits the same line as Developer B
* GitHub blocks merge
* A resolves conflict via the web editor
* PR merges successfully

# **11. MODULE 10 ‚Äî GIT WORKFLOWS**

## **1. Git Flow**

Branches:

* main
* develop
* feature/*
* release/*
* hotfix/*

## **2. GitHub Flow**

* Always branch from main
* Frequent merges via PRs
* Ideal for CI/CD

## **3. Trunk-Based Development**

* Small, frequent commits
* Feature flags

### **Diagram (text):**

```
[main] ‚Üê small branches ‚Üê merged daily
```

## **Sample Questions:**

### **1. What is the difference between fetch and pull?**

A) fetch merges automatically
B) fetch downloads only
C) pull downloads and merges
Correct: B & C

### **2. Which command shows HEAD history?**

* `git reflog`

### **3. Which merge strategy creates a single commit that summarizes the branch?**

* Squash merge

### **4. What command discards uncommitted file changes?**

* `git restore <file>`

### **5. What is the purpose of a pull request?**

* Code review, discussion, automated checks, and merging
